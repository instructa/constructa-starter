Below is a **full architecture & implementation plan** for migrating the **Constructa Starter** codebase from *PostgreSQL + Drizzle* to **Convex**, integrating the **Convex ⇄ TanStack Query** adapter, replacing the Drizzle Better‑Auth adapter with **convex‑better‑auth**, and shipping a working **“Macro Calculator”** feature that demonstrates live Convex queries and mutations.

---

## 0 — High‑level design decisions

| Area                 | Decision                                                                                           | Rationale                                                                                                        |
| -------------------- | -------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **DB / persistence** | Use **Convex cloud** (or self‑host) as the single persistent tier.                                 | Eliminates manual schema migrations, fits Constructa’s “AI‑first” DX, and provides automatic live‑query updates. |
| **Data‑fetching**    | Keep **TanStack Query** as the cache layer but wire it to Convex with **@convex‑dev/react‑query**. | Gives reactive Convex subscriptions *and* keeps existing <code>QueryClient</code> API. ([docs.convex.dev][1])    |
| **Authentication**   | Swap <code>drizzleAdapter</code> for **@better‑auth‑kit/convex** (aka *convex‑better‑auth*)        | Maintains Better‑Auth API while persisting auth objects inside Convex. ([github.com][2], [docs.convex.dev][3])   |
| **Product rename**   | Rename npm package & docs to **constructa‑convex**.                                                | Signals DB change and avoids confusion.                                                                          |
| **Example feature**  | Build “**Macro Calculator**” (per‑day macro log) to prove live data flow.                          | Simple schema; shows queries, mutations, auth gating, optimistic UI.                                             |


Create **`src/lib/convex-query-client.ts`**:

```ts
import { ConvexQueryClient } from "@convex-dev/react-query";
import { convex } from "./convex-client";
import { QueryClient } from "@tanstack/react-query";

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      queryKeyHashFn: new ConvexQueryClient(convex).hashFn(),
      queryFn: new ConvexQueryClient(convex).queryFn(),
    },
  },
});

export const convexQueryClient = new ConvexQueryClient(convex);
convexQueryClient.connect(queryClient);
```

## 5 — Convex data model

### 5.1 `/convex/schema.ts`

```ts
import { defineSchema, v } from "convex/server";

export default defineSchema({
  macros: {
    userId: v.string(),
    date: v.string(),       // YYYY‑MM‑DD
    protein: v.number(),
    carbs: v.number(),
    fat: v.number(),
    calories: v.number(),   // redundant but useful
    createdAt: v.number(),
  },
  // plus Better‑Auth tables auto‑generated by convex-better-auth
});
```

Include secondary index `(userId, date)` for fast per‑day lookups.

### 5.2 Convex functions

`/convex/macros/*`

```ts
// query listMacros.ts
import { query } from "convex/server";
export const listMacros = query({
  args: { date: v.string() },
  handler: async (ctx, { date }) => {
    const user = await ctx.auth.getUser();
    if (!user) throw new Error("unauth");
    return ctx.db
      .query("macros")
      .withIndex("by_userId_date", (q) =>
        q.eq("userId", user.subject).eq("date", date)
      )
      .collect();
  },
});

// mutation upsertMacro.ts
import { mutation } from "convex/server";
export const upsertMacro = mutation({
  args: {
    date: v.string(),
    protein: v.number(),
    carbs: v.number(),
    fat: v.number(),
  },
  handler: async (ctx, input) => {
    const user = await ctx.auth.getUser();
    if (!user) throw new Error("unauth");
    const calories = input.protein * 4 + input.carbs * 4 + input.fat * 9;
    await ctx.db.insert("macros", {
      userId: user.subject,
      ...input,
      calories,
      createdAt: Date.now(),
    });
  },
});
```

## 9 — Potential side effects & mitigations

| Impact                   | Details                                                                                       | Mitigation                                                                                                                                                    |
| ------------------------ | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Cold‑start latency**   | First Convex function call incurs ≈ 100–200 ms.                                               | Acceptable for SaaS; can pre‑warm with loader queries.                                                                                                        |
| **Transactional limits** | Convex enforces function runtime & size limits.                                               | Macro use‑case trivial; audit future file‑upload features.                                                                                                    |
| **Auth plugin parity**   | Some Better‑Auth plugins (rate‑limit, magic‑link) not yet ported in convex‑better‑auth alpha. | Verify each required plugin; open GitHub issues as needed.                                                                                                    |
| **SSR query prefetch**   | Convex live queries in loaders must be serialized carefully.                                  | TanStack Start loader can return `await serverQuery(...)` then hydrate on client – see Convex docs “Loaders” example. ([convex-tanstack-start.vercel.app][4]) |
